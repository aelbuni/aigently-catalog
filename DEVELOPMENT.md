# Aigently Catalog — Developer Guide

Everything you need to run the MCP server locally, refresh catalog data from the pipeline, and connect it to Cursor or Claude Code.

---

## What this repo contains

```
packages/
  catalog-data/      ← Static JSON files consumed by the MCP server
    stacks.json        - 11 tech stacks
    rules.json         - Security rules with bodyMdx + layer mappings
    threats.json       - 500+ CVEs with AI-generated guardrail patterns
    guardrails.json    - Pre-synthesized guardrail content per (stack, layer)
    manifest.json      - Generation timestamp and counts

  mcp-server/        ← The MCP server that IDEs connect to
    src/               - TypeScript source
    dist/              - Compiled output (committed, ready to run)
    scripts/           - Test and utility scripts
```

The `catalog-data/` files are exported daily from the Aigently pipeline database
and committed to this repo. The MCP server reads them at startup — **no network
calls, no database, no API key required**.

## Running the MCP server locally

### Step 1 — Install dependencies

```bash
cd aigently-catalog
npm install
```

### Step 2 — Build the server

```bash
npm run build -w @aigently/mcp-server
```

The compiled output lands in `packages/mcp-server/dist/`. The `dist/` directory
is committed, so if you only want to run the existing version without rebuilding,
you can skip this step.

### Step 3 — Verify it starts

Send a JSON-RPC request via stdin:

```bash
echo '{"jsonrpc":"2.0","id":1,"method":"tools/list"}' \
  | node packages/mcp-server/dist/index.js
```

You should see a JSON response listing 8 tools. If you see an error about a
missing catalog file, see [Refreshing catalog data](#refreshing-catalog-data).

### Step 4 — Run the smoke tests

```bash
node packages/mcp-server/scripts/test-tools.mjs
```

All 11 tests should pass in under 5 seconds.

## Development workflow

### Watch mode (auto-recompile on change)

```bash
npm run dev -w @aigently/mcp-server
```

TypeScript will recompile whenever you save a source file. You do not need to
restart the MCP server in your IDE after a recompile — most IDEs restart it
automatically when the process exits or the binary changes.

### Testing a single tool manually

```bash
echo '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{
  "name": "get_security_context",
  "arguments": {
    "intent": "building a login API in Next.js with JWT",
    "file_path": "app/api/auth/route.ts"
  }
}}' | node packages/mcp-server/dist/index.js | python3 -m json.tool
```

## Refreshing catalog data

The `catalog-data/*.json` files are generated by the pipeline scripts in this repo
(`pipeline/scripts/export-catalog.ts`). They are committed here and auto-updated
by the daily CI/CD workflow.

To refresh them manually:

```bash
# Fetch latest CVEs and regenerate catalog JSON (in this repo)
npm run sync:threats        # fetch from NVD, CISA KEV, GHSA, OSV, npm
npm run amplify:threats     # AI-generate patterns for new threats
npm run summarize:rules     # AI-cluster CVEs into summaries
npm run export:catalog      # write JSON to packages/catalog-data/
git add packages/catalog-data/ && git commit -m "chore(catalog): manual refresh"
```

Requires `ANTHROPIC_API_KEY` (for amplify + summarize) and `DATABASE_URL` in `pipeline/.env`.
See [`pipeline/.env.example`](pipeline/.env.example) for all variables.

### What each JSON file contains

| File | Updated by | Description |
|------|-----------|-------------|
| `stacks.json` | `export:catalog` | Stack slugs, names, ecosystems |
| `rules.json` | `export:catalog` | Security rules with bodyMdx and layer assignments |
| `threats.json` | `export:catalog` | CVEs with AI amplification patterns and stack/layer assignments |
| `guardrails.json` | `export:catalog` | Pre-synthesized guardrail blocks per stack×layer |
| `manifest.json` | `export:catalog` | Version, timestamp, counts |

## Available tools

| Tool | Input | What it returns |
|------|-------|-----------------|
| `get_security_context` | `intent`, `file_path?`, `stacks?` | Matching rules + top CVEs for the detected stack |
| `compose_guardrail` | `stack_slug`, `intent?`, `layer_slugs?`, `target_ide?`, `rule_type?` | Complete IDE-ready guardrail file |
| `list_stacks` | — | All 11 stacks |
| `list_layers` | — | All 15 protection layers |
| `get_rule` | `slug` | Full rule body + summary |
| `search_threats` | `query?`, `severity?`, `owasp_ref?`, `stack_slug?`, `layer_slug?`, `limit?` | Filtered CVE list |
| `get_threat` | `id` (CVE ID or publicId) | Full threat detail + AI patterns |
| `get_manifest` | — | Catalog version and counts |

### `target_ide` values for `compose_guardrail`

| Value | Output file | IDE |
|-------|-------------|-----|
| `cursor` | `.cursor/rules/aigently-{stack}-security.mdc` | Cursor |
| `claude-code` | `CLAUDE.md` | Claude Code |
| `windsurf` | `.windsurfrules` | Windsurf |
| `copilot` | `.github/copilot-instructions.md` | GitHub Copilot |
| `cline` | `.clinerules` | Cline |

### `rule_type` values for `compose_guardrail`

| Value | Includes |
|-------|---------|
| `all` (default) | Both pattern rules and dependency advisory rules |
| `pattern` | Only coding-pattern rules (ALWAYS/NEVER directives) |
| `deps` | Only dependency version advisory rules |

## Connecting to an IDE

All IDEs use the same JSON structure. The only difference is the `AIGENTLY_TARGET_IDE`
env var — set it once and the server uses it as the default for all tool calls.

Get your server path:

```bash
echo "$(pwd)/packages/mcp-server/dist/index.js"
```

### Cursor — `.cursor/mcp.json`

```json
{
  "mcpServers": {
    "aigently": {
      "command": "node",
      "args": ["/absolute/path/to/packages/mcp-server/dist/index.js"],
      "env": { "AIGENTLY_TARGET_IDE": "cursor" }
    }
  }
}
```

### Claude Code — `~/.claude/settings.json`

```json
{
  "mcpServers": {
    "aigently": {
      "command": "node",
      "args": ["/absolute/path/to/packages/mcp-server/dist/index.js"],
      "env": { "AIGENTLY_TARGET_IDE": "claude-code" }
    }
  }
}
```

### Windsurf, Copilot, Cline

Same structure — set `AIGENTLY_TARGET_IDE` to `windsurf`, `copilot`, or `cline`.
See [packages/mcp-server/README.md](packages/mcp-server/README.md) for IDE-specific
config file locations.

> **Why the env var?** The LLM inside any IDE doesn't inherently know which IDE it's
> running in. Rather than requiring every tool call to pass `target_ide`, we configure
> it once at the server level. Users set it once in `mcp.json` and never think about
> it again.

## Troubleshooting

### "Catalog file not found"

The server cannot read `packages/catalog-data/stacks.json`. Run:

```bash
ls packages/catalog-data/
```

If the files are missing, regenerate them from the pipeline (see
[Refreshing catalog data](#refreshing-catalog-data)).

### "Cannot find module"

The `dist/` directory is missing or stale. Rebuild:

```bash
npm run build -w @aigently/mcp-server
```

### Cursor shows "MCP server failed to start"

Check that Node.js 18+ is on your `PATH`:

```bash
which node && node --version
```

Use the absolute path in `mcp.json` rather than `npx` to avoid PATH issues in
sandboxed IDE environments.

### Server starts but tools return empty results

The catalog data may be stale or filtered down. Check the manifest:

```bash
echo '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"get_manifest","arguments":{}}}' \
  | node packages/mcp-server/dist/index.js | python3 -m json.tool
```

Look at `counts.threats`, `counts.rules`, and `counts.guardrails`. If any are 0,
re-run the export pipeline.
